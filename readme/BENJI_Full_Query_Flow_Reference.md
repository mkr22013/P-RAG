# BENJI — Full Query Flow Reference

How a member query travels from raw input to final response.
Covers Medical, Dental, Vision, and Rx paths in detail — including
category detection, topic resolution, index scoring, chunk filtering,
and response building.

---

## Architecture Overview

```
Member Query
     │
     ▼
_preprocess_query()          ← sanitize, detect pasted responses / greetings
     │
     ▼
_detect_and_route()          ← category detection, topic resolution, cache key
     │
     ├─ p_type = "rx"   ──►  _handle_rx_query()    ← dual-index Rx path
     │
     └─ p_type = medical/dental/vision
               │
               ▼
          _retrieve_and_cache()     ← cache hit OR fresh index fetch
               │
               ▼
          _build_response()         ← parser → mini-LLM → full LLM
               │
               ▼
          Final Response: {"answer": str, "pages": list, "source": str}
```

**Core principle:** LLM is called ONLY when rules fail. 51% of queries use
zero tokens. Cost amounts are NEVER generated by LLM — extracted directly
from structured index.

## Step 1 — `_preprocess_query()`

**File:** `main/client.py`

### What It Does

Sanitizes input and handles two early-exit cases before any RAG pipeline:

**Pasted response detection:**
Member copies back our own drug list → we detect this and redirect:
```python
_pasted_response = (
    "ask me about any specific drug" in query_lower
    or query_lower.strip().startswith("here are the medications")
)
```

**Conversational intent (`is_conversational()`):**
Greetings, follow-ups, non-benefit questions → return guidance immediately.
`is_conversational()` is in `utility/category.py` — checks for short queries,
common greetings, and queries with no insurance signal words.

### Output
```python
{
    "query_lower": str,      # lowercased query
    "query_words": list,     # cleaned word tokens (re.sub punctuation)
    "early_return": bool,
    "response": dict | None
}
```

## Step 2 — `_detect_and_route()`

**File:** `main/client.py`

### 2a. Category Detection

**Function:** `detect_category(query_words, query)` in `utility/category.py`

Three-stage detection, first match wins:

#### Stage 1 — Drug name check (`is_drug_name_query`)
Checks if any word in the query is a known drug name by looking it up in
`drug_words.json`. If found → `"rx"`.

```python
# drug_words.json loaded with 48h TTL cache
# is_drug_name_query() checks query_words against the word set
if is_drug_name_query(query_words):
    return "rx"
```

#### Stage 2 — Rule-based signals (`detect_category_rule_based`)
Checks for explicit category signal words:

```python
# Dental signals: "dental", "tooth", "crown", "filling", "canal", etc.
# Vision signals: "vision", "eye", "glasses", "contacts", etc.
# Rx signals: "drug", "prescription", "formulary", "tier", "pharmacy"
# Medical signals: "medical", "hospital", "copay", "deductible", "specialist",
#                  "chiropractic", "dialysis", "bariatric", "outofpocket", etc.
```

No fuzzy matching here — exact word match only. Safer for category routing.

#### Stage 3 — LLM fallback (`detect_category` LLM path)
When rules find nothing, one small LLM call (~200 tokens):
```json
{"category": "medical"}
```

Special LLM prompt rule: illness/condition queries without service words
route to `"rx"` so members asking "what is covered for asthma?" get drug
results rather than medical benefit results.

#### History fallback
If category still not found → `detect_category_from_history()` scans last
10 messages for category signals.

### 2b. Topic Resolution

**Function:** `resolve_insurance_topic(query_words, query_lower, p_type)` in
`utility/topic_resolver.py`

Routes to one of four category-specific resolvers:

```python
if p_type == "dental":   return resolve_dental_topic(...)
elif p_type == "vision": return resolve_vision_topic(...)
elif p_type == "rx":     return resolve_rx_topic(...)
else:                    return resolve_medical_topic(...)
```

#### Why Four Separate Resolvers

`smart_match()` uses fuzzy matching (0.8 threshold) for typo tolerance.
This is safe for short, specific dental/vision terms but dangerous for
medical terms — `"vasectomy"` fuzzy-matched `"reconstruction"` causing
wrong chunks to be returned.

```
resolve_medical_topic()   ← _exact() match only (no fuzzy)
resolve_dental_topic()    ← smart_match OK (short dental terms, typo tolerance)
resolve_vision_topic()    ← smart_match OK (short vision terms)
resolve_rx_topic()        ← minimal — Rx is keyword-driven
```

#### `_detect_emergency()` — Shared Helper
Emergency detection runs for ALL four resolvers. Previously only ran in
the medical section — splitting into four functions caused dental emergency
queries (D9440 after-hours copay) to lose the `"emergency"` keyword and
return wrong chunks.

```python
def _detect_emergency(query_words, query_lower, topics, add_keyword):
    # fires for: medical ER, dental after-hours (D9440), vision eye injury
    if smart_match("emergency", ...):
        topics.append("emergency")
        add_keyword("emergency room")
        add_keyword("emergency")
        add_keyword("after hours")  # needed for D9440 dental chunk
```

#### Medical Topic Resolution (`resolve_medical_topic`)

Uses `_exact(word, query_words)` — exact word match, no fuzzy:

```
Section 1:  Deductible / OOP       → smart_match OK (short safe terms)
Section 2:  Emergency              → _detect_emergency() shared helper
Section 3:  Diagnostic / Imaging   → smart_match OK
Section 4:  Network                → smart_match OK
Section 7:  PCP / Specialist       → smart_match OK
Section 8:  Mental Health          → smart_match + regex
Section 9:  Maternity              → _exact()
Section 10: Hospital               → _exact()
Section 12: Pharmacy               → _exact()
Section 13: Rehabilitation         → _exact() (chiropractic, acupuncture added)
Section 14: Other flags            → _exact()
Sections 15-32: New medical topics → _exact() ONLY
    (allergy, immunotherapy, therapeutic, transplants, virtual care,
     nicotine, dialysis, foot care, home health, breast reconstruction,
     gender affirming, newborn, clinical trials, transportation,
     bariatric, out-of-pocket, preventive care)
```

#### Residual Keyword Extraction

After topic detection, `_build_result()` runs residual extraction — any
query word with 4+ characters not already in topics/keywords gets added.
This catches words not covered by patterns (e.g. "vasectomy") so they
can still influence chunk scoring.

#### LLM Topic Fallback

When topic resolver returns empty topics AND category is not `"rx"`:
```python
if not found_topics and p_type != "rx":
    found_topics, keywords = _extract_topics_via_llm(query, ...)
```
LLM returns `{"topics": [...], "keywords": [...]}`. Topics are cleaned:
- Skip if > 2 words (too noisy)
- Skip if all words are noise words
- Skip if equals category name ("medical", "dental", "vision")

#### History Topic Recovery

When current query has no topic (e.g. "how much does it cost?" follow-up):
```python
_recover_topics_from_history(history, p_type, found_topics, keywords)
```
Scans last 10 user messages, stops at category boundary (dental → medical).

### 2c. Cache Key

```python
cache_key = f"{p_type}_{topics_sorted}_{keywords_sorted}"
# Example: "medical_rehabilitation_chiropractic_rehabilitation therapy"
```

Sorted so "copay deductible" and "deductible copay" produce the same key.

## Step 3 — Rx Path: `_handle_rx_query()`

**File:** `main/client.py`

The Rx path is completely separate from medical/dental/vision. It uses a
dual-index approach — formulary index for drug coverage + medical index
for tier costs.

### Full Rx Flow

```
query_words → rx_keywords (filter noise words)
           → spelling correction (correct_drug_spelling)
           │
           ├─ is_drug_name_query() ?
           │      NO → condition_resolver.resolve_query_to_drugs()
           │              → condition_drugs list
           │
           ├─ condition_drugs > 10 ?
           │      YES → _build_condition_drug_list_response()
           │            (name list + "ask me about one drug")
           │
           ├─ condition_drugs ≤ 10 ?
           │      YES → use condition_drugs as search_terms
           │
           └─ is_drug_name_query() ?
                  YES → use rx_keywords as search_terms
```

### Step Rx-1: Drug Coverage from Rx Formulary Index

```python
rx_result = query_insurance_benefits(
    query=query,
    topics=search_terms,       # drug names or condition drug list
    category="rx",
    keywords=condition_keywords,
    include_requirements_text=not query_has_real_drug_name
)
```

#### Three Filter Strategies (in `_filter_rx_results()`)

**Strategy 1 — Generic fallback** (no drug name, no concept word):
Strip RX section entirely — results are arbitrary drugs, not what member asked.
Keep only INFO section for LLM synthesis.

**Strategy 2 — Condition drugs** (≤10 matching drugs):
Keep only entries where `drug_name` first word is in condition_drugs_set.
Separate into covered vs not-covered for response builder.

**Strategy 3 — Drug name match** (real drug name in query):
Keep only entries where `drug_name` fuzzy-matches query keywords via
`is_drug_match()`. This handles: exact match, character similarity,
phonetic matching (e.g. "humera" → "humira").

#### Drug Not Found Response

When no RX or INFO sections remain after filtering:
```python
return _build_drug_not_found_response(rx_keywords, member_info)
# "Humera was not found on your formulary. It may be known by a different name..."
```

### Step Rx-2: Tier Cost from Medical Index

```python
cost_result = query_insurance_benefits(
    query="prescription drug tier cost",
    topics=("prescription drug",),
    category="medical",  # or "sbc" for HSA plans
    keywords=("prescription", "drug", "generic", "brand", "specialty", "tier")
)
```

### Step Rx-3: Build Response

```python
answer, rx_pages, cost_pages = build_rx_response(
    rx_context=rx_result,
    cost_context=cost_result
)
```

`build_rx_response()` in `utility/response_builder.py` builds:
- Drug Coverage table (tier, status, requirements)
- Your Cost table (tier name → copay/coinsurance amounts)
- NOT_COVERED section (drugs in query but not on formulary)

## Step 4 — Medical/Dental/Vision: `_retrieve_and_cache()`

**File:** `main/client.py`

### Cache Check

```python
cached_context = await get_query_result(cache_key)
if cached_context:
    return cached_context  # 0 tokens, instant
```

Cache TTL: 24 hours. Key includes category + topics + keywords (sorted).

### Fresh Fetch: `_handle_direct_query()`

```python
tool_result = query_insurance_benefits(
    query=query,
    topics=tuple(found_topics),
    category=p_type,
    keywords=tuple(keywords),
    member_info=json.dumps(member_info),
    include_limitations_text=include_limitations
)
```

**Cost signal detection:**
```python
_COST_SIGNAL_WORDS = {"cost", "copay", "coinsurance", "much", "deductible", ...}
is_cost_specific_query = any(w in query_words for w in _COST_SIGNAL_WORDS)
include_limitations = not is_cost_specific_query
```

Cost queries don't search limitations text (avoids noise). Conceptual
queries DO search it (e.g. "is X covered when travelling" lives in limitations).

### Inside `query_insurance_benefits()` — Chunk Scoring

**File:** `insurance_mcp/tools.py`

This is the core retrieval engine. For each topic:

#### Scoring Formula

```
chunk_score = topic_score + keyword_score + content_score

topic_score:    HIGH   — chunk's topic/event name matches query topic
keyword_score:  MEDIUM — chunk's chunk_keywords overlap with query keywords
content_score:  LOW    — raw text content contains query words
```

#### chunk_keywords Quality (July 2026 fix)

`get_smart_keywords()` in `utility/utils.py` now uses domain-specific
patterns instead of generic 7-char word extraction:

```
Before: ['in_network', 'out_of_network', 'limitations', 'service', 'deductible']
        (JSON field names leaked in — all chunks scored equally)

After:  ['hospital', 'inpatient', 'deductible', 'coinsurance']
        (domain patterns — Hospital chunk scores higher than Hospice chunk
         for "inpatient hospital stay" query)
```

30+ domain patterns: hospital, inpatient, outpatient, hospice, surgical,
transplant, maternity, newborn, dialysis, rehabilitation, skilled nursing,
home health, pcp, specialist, emergency, urgent-care, copay, deductible,
coinsurance, out-of-pocket, prior-auth, pharmacy, dental, vision,
imaging, diagnostic, mental-health, therapy, preventive, allergy,
gender-affirming, bariatric, clinical-trials, transportation, etc.

#### Member-Level Access Control

```python
# member_info filters index to EXACT member plan documents
# Member from Group 1000016 Plan A CANNOT get Plan B results
# Structural guarantee — DB query constraint, not policy rule
SELECT sub_index_path FROM master_index
WHERE group_number = ? AND plan = ? AND variant = ?
```

#### Result Selection

```python
FINAL_SELECTED_COUNT = min(TOP_K, total_matched)
# Typically TOP_K = 10-12 for cost queries
# Chunks sorted by score, top-K returned
```

### Keyword Filtering (`_filter_by_keywords()`)

After retrieval, post-filter by keywords:

```python
# COST and INFO sections: NEVER filtered (always keep)
# Other sections: keep only chunks containing at least one keyword
if current_header in ("### SECTION: COST", "### SECTION: INFO"):
    rebuilt.append(block)  # always keep
    continue
if any(k.lower() in block.lower() for k in keywords):
    rebuilt.append(block)  # keep if keyword match
```

### Context Trimming (`_trim_context()`)

```python
max_chars = 8000
# Breaks at paragraph boundary rather than mid-sentence
```

### Cache Write

```python
await set_query_result(cache_key, clean_context)  # 24h TTL
```

## Step 5 — `_build_response()`

**File:** `main/client.py`

Three-tier response building. Always tries cheapest tier first.

### Section Detection

```python
if "### SECTION: COST" in clean_context: sections.append("cost")
if "### SECTION: INFO" in clean_context: sections.append("info")
```

---

### Tier 1 — Parser (0 tokens, fastest)

#### COST + INFO (most common)

```python
cost_table, cost_pages = build_cost_table(cost_ctx, query, keywords, found_topics)
info_table, info_pages = build_info_response(info_ctx, query, keywords)
```

**`build_cost_table()`** in `utility/response_builder.py`:
- Parses structured JSON cost chunks
- Builds markdown table: Benefit | Service | In-Network | Out-of-Network | Limitations
- Multi-event detection: if chunks span multiple events, shows all
- Noise detection: if too many unrelated events → returns `"__USE_LLM__"`
- Exclusion query suppression: `"dental exclusions"` topic → suppress cost table
- Pure dental cost query suppression: radiographic/bitewing/etc. → suppress info

**`build_info_response()`** in `utility/response_builder.py`:
- Parses INFO chunks (coverage policy text)
- Builds markdown table: Topic | Coverage Information
- Returns `"__USE_LLM__"` if content seems unrelated to query

**Combined response:**
```python
summary = generate_summary(query, info_table=info_table, keywords=keywords)
combined = cost_table + "\n\n" + info_table
answer = f"{summary}\n\n{combined}" if summary else combined
```

**`generate_summary()`:**
- One LLM call, max 40 tokens
- Skipped for value-seeking queries ("how much", "what is my copay")
- Skipped if info content not relevant to query (keyword relevance check)
- NEVER includes dollar amounts or percentages (cost table handles those)

#### COST only / INFO only

If only one section detected, builds just that table.

---

### Tier 2 — Mini-LLM (~50-100 tokens)

When parser fails (noisy/mixed context):

```python
mini_prompt = f"Extract: (1) benefit name as in benefits booklet, (2) intent: cost/info/both"
# Returns: {"benefit": "Rehabilitation Therapy", "intent": "cost"}
```

Re-runs parser with the canonical benefit name → more precise scoring.

---

### Tier 3 — Full LLM (~500-1500 tokens)

Last resort when Tier 1 and 2 both fail:

```python
final_messages = [
    {"role": "system", "content": generate_ironclad_instruction()},
    {"role": "user", "content": f"Question: {query}\nContext: {clean_context}"}
]
final_answer = llm_chat(messages=final_messages, temperature=0.0)
```

**Iron curtain post-processing:**
```python
# Strip to table content only — removes any LLM preamble
if "|" in final_answer:
    start = final_answer.find("|")
    end = final_answer.rfind("|")
    final_answer = final_answer[start:end+1].strip()
```

**Page extraction from context:**
```python
# Extract page numbers from all chunks used by Tier 3
for item in re.split(r"Item \d+:", clean_context):
    page_match = re.search(r'"page_number"\s*:\s*(\d+)', item)
    if page_match:
        t3_pages.append(int(page_match.group(1)) - page_offset)
```

## Response Shape

All paths return:

```python
{
    "answer": str,          # markdown table(s) + optional summary sentence
    "pages": list[int],     # page numbers in plan document (offset-adjusted)
    "source": str,          # plan name + page range
    "category": str         # optional — medical/dental/vision
}
```

### Page Offset

Each plan has a `page_offset` in member_info. PDF page numbers are adjusted:
```python
def _add_offset(raw_pages):
    return sorted(p - page_offset for p in raw_pages if p > page_offset)
```

```
Medical:  "Premera Employees Health Plan – Standard PPO Retiree Plan | Pages 14, 20"
Rx:       "Essentials Formulary Drug List (E4) | Pages 39, 104 ||
           Premera Employees Health Plan | Page 18"
```

## Error Handling

```python
try:
    # full pipeline
except Exception as e:
    print(f"❌ SYNTHESIS ERROR: {e}")
    return {"answer": f"⚠️ Client Logic Error: {str(e)}", "pages": [], "source": ""}
```

Retrieval failure returns `"RETRIEVAL ERROR"` string — caught by `_retrieve_and_cache()`
which returns empty context → guidance message returned to member.

## Key Design Decisions

**Why exact match for medical topics:**
`smart_match` fuzzy threshold 0.8 caused "vasectomy" to fire on "reconstruction"
topic in topic_resolver. Medical terms are long and specific enough that exact
match is sufficient and much safer.

**Why COST and INFO sections are never filtered:**
Post-retrieval keyword filtering could accidentally remove the correct cost row.
COST and INFO are always kept — noise is handled by the response builder's
multi-event detection and noise threshold.

**Why cost amounts are never generated by LLM:**
`build_cost_table()` extracts amounts directly from structured index JSON.
The LLM sees the final answer string (already a table), not the raw cost data.
Hallucination on dollar amounts is structurally impossible.

**Why topic resolver has 4 functions:**
Before the split, all sections ran for every query regardless of category.
This meant a dental emergency query also ran medical sections, inadvertently
adding keywords that helped score the D9440 chunk. The split broke this.
Solution: `_detect_emergency()` shared helper called by all four resolvers.
