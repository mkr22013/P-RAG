# BENJI — Benefit Exploration & Navigation with Just-in-time Intelligence

## Executive Summary

BENJI is a production-grade Retrieval-Augmented Generation (RAG) system purpose-built for insurance benefit queries. It achieves **100% numerical accuracy** on dollar amounts and coinsurance values at **95% lower LLM token cost** compared to traditional RAG architectures.

---

## The Problem with Traditional RAG

Traditional RAG systems send every query through a full LLM pipeline:

```
Query → Embed → Vector Search → LLM Rerank → LLM Synthesis
```

**Problems:**
- Every step costs tokens — 1,500 to 5,000 tokens per query
- LLM can hallucinate dollar amounts, copays, coinsurance values
- 70-85% accuracy on structured financial data
- Expensive at scale (national account = millions of queries/year)
- No auditability — hard to trace where an answer came from

---

## BENJI Architecture: Minimum LLM, Maximum Accuracy

BENJI uses a **3-tier pipeline** that minimizes LLM calls while maximizing accuracy on structured benefit data.

```
Member Query
     ↓
┌─────────────────────────────────────────────────────┐
│  TIER 0: Rule-Based (0 tokens)                      │
│  - Category detection (medical/dental/vision/rx)    │
│  - Topic resolution (emergency room, prophylaxis)   │
│  - Direct retrieval from indexed plan documents     │
│  - Structured table parser → exact dollar amounts   │
│                                                     │
│  Handles: ~85% of all queries                       │
└─────────────────────────────────────────────────────┘
     ↓ (if rules fail)
┌─────────────────────────────────────────────────────┐
│  TIER 1: Mini-LLM (30-100 tokens)                   │
│  - Category fallback: "medical" / "dental" / "rx"   │
│  - Topic fallback: ["emergency room"] / ["crown"]   │
│  - Input: ~50 tokens, Output: single word/phrase    │
│                                                     │
│  Handles: ~10% of all queries                       │
└─────────────────────────────────────────────────────┘
     ↓ (if context is complex/noisy)
┌─────────────────────────────────────────────────────┐
│  TIER 2: Full LLM Synthesis (200-500 tokens)        │
│  - Natural language summary of structured data      │
│  - LLM sees pre-verified chunks, never raw text     │
│  - Numbers/amounts are NEVER generated by LLM       │
│                                                     │
│  Handles: ~5% of all queries                        │
└─────────────────────────────────────────────────────┘
```

## Why 100% Numerical Accuracy

The key design principle: **LLMs never generate numbers.**

```
Traditional RAG:
  LLM reads raw PDF chunks → generates answer with numbers
  Risk: LLM can confabulate "$350" as "$250" or omit copay waiver conditions

BENJI:
  Rule-based scoring → extracts exact rows from pre-indexed structured data
  LLM receives pre-verified structured chunks → synthesizes natural language only
  Dollar amounts, copays, coinsurance → always from source document, never generated
```

## Architecture Components

### Document Indexing Pipeline

```
PDF Booklets (Medical, Dental, Vision, Rx, SBC)
     ↓
classify_document()   — LLM identifies plan metadata from document
     ↓
generate_sub_index()  — Parses PDF into structured chunks
     ↓
JSON index file       — Stored in Azure Blob Storage
     ↓
PostgreSQL master index — Plan lookup table (year, group, variant)
     ↓
Redis cache           — Instant retrieval, event-driven invalidation
```

### Query Pipeline

```
Member Query
     ↓
is_conversational()           — Detects greetings/non-benefit queries (0 tokens)
     ↓
detect_category()             — Rule-based: medical/dental/vision/rx (0 tokens)
  └─ LLM fallback             — Only when rules fail (~100 tokens)
     ↓
resolve_insurance_topic()     — Rule-based: emergency room, prophylaxis, crown (0 tokens)
  └─ LLM fallback             — Only when rules fail (~150 tokens)
     ↓
get_plan_data_from_disk()     — Direct call, no LLM tool overhead (0 tokens)
  → Redis cache hit           — Sub-millisecond chunk retrieval
  → Redis miss → Blob download → Redis cache set
     ↓
score_chunks()                — Rule-based scoring: topic match + keyword match (0 tokens)
     ↓
build_cost_table()            — Structured parser → exact markdown table (0 tokens)
  └─ LLM synthesis            — Only for complex/narrative queries (~300 tokens)
     ↓
Final response                — Structured table with page references
```

### Infrastructure (Azure-Native)

```
Azure Blob Storage    — PDF source documents + JSON index files
Azure PostgreSQL      — Master plan index (year/group/variant → blob path)
Azure Redis Cache     — Event-driven cache, invalidated on re-index
Azure Service Bus     — Cache invalidation messages
Azure APIM            — Auth gateway (Auth0 JWT validation)
Azure Container Apps  — Application hosting
```

## Direct Category Routing (No LLM Tool Call)

A key optimization — categories with well-defined indices bypass LLM tool orchestration entirely:

```python
# client.py
DIRECT_CATEGORIES = {"medical", "dental", "vision", "rx"}

if category in DIRECT_CATEGORIES:
    # Zero LLM overhead — direct function call
    result = get_plan_data_from_disk(query, topics, category, keywords)
else:
    # Future categories (claims, authorizations) use LLM orchestration
    result = llm_tool_call(query, tools)
```

**Savings per medical/dental/vision/rx query:** ~200 tokens (eliminated tool decision LLM call)

### How Baselines Are Established

1. Queries run against real API (`POST /chat`) — same path as UI
2. Responses saved as JSON baselines including answer, page references, source plan
3. Future runs compare dollar amounts exactly — any change = **HARD FAIL**
4. Natural language changes reviewed and approved/rejected interactively

```bash
# Verify all categories against baselines
python -m tests.golden_test --verify

# Verify single category
python -m tests.golden_test --verify --category medical

# CI/CD mode — non-interactive, fails on any diff
python -m tests.golden_test --verify --ci
```

### What the Golden Test Validates

```
HARD FAIL (blocks deployment):
  - Any dollar amount changes      e.g. $350 → $250
  - Any coinsurance changes        e.g. 20% → 25%
  - Any copay changes              e.g. $25 copay → $30 copay

REVIEW REQUIRED:
  - Natural language summary changes
  - Page reference changes
  - New rows added/removed

AUTO-PASS:
  - Whitespace/formatting changes
  - Equivalent phrasing
```

## Security

- **Auth0 JWT validation** on every API request
- Audience + signature + expiry validated before any endpoint is reached
- `/health` only endpoint excluded from auth

- Same endpoint serves React UI (member tokens) and external agents (M2M tokens)

## Multi-Agent Integration

External agent teams can call the same `/chat` endpoint:

```
POST /chat
Authorization: Bearer <m2m_token>

prompt=is bariatric surgery covered?
member_key=ABC123
group_number=1000016
```

Returns the same final vetted response as the UI — no raw chunks, no post-processing needed by the calling agent.

Or via MCP protocol:
```python
# query_benefits MCP tool — returns final vetted response
result = await mcp_client.call_tool("query_benefits", {
    "query": "is bariatric surgery covered?",
    "member_key": "ABC123",
    "group_number": "1000016"
})
```
